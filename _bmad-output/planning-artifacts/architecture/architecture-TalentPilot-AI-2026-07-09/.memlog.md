- (decision) AD-1 Paradigm: Three-Tier Web Application with CQRS-lite separation. Client (React SPA) → API (FastAPI) → Database (PostgreSQL). Read paths optimized separately from write paths for watch-progress conditional writes and dashboard real-time updates.
- (decision) AD-2 Module Boundaries: Domain-driven modules with clear ownership. auth (Session/JWT), assignments (HR workflow), content_discovery (RAG/vector search), watch_progress (capture/resume), dashboard (aggregation/real-time). Each module owns its database tables, service logic, and API routes.
- (decision) AD-3 Data Ownership: assignments module owns Assignment records (Employee+Skill+Status). watch_progress module owns skill_progress table (video position/timestamp). content_discovery owns content_catalog + embeddings. No shared mutable state across module boundaries - communication via service interfaces only.
- (decision) AD-4 Conditional Write Semantics: watch_progress writes use event-timestamp ordering, not position magnitude. Server validates: (1) incoming timestamp > stored timestamp, (2) position advance rate is plausible (anti-spoofing), (3) valid session ties to actual Assignment. Stale out-of-order writes are rejected; legitimate rewinds (newer timestamp, lower position) are accepted. Implements FR-7 NFR.
- (decision) AD-5 Coaching-Only Access Enforcement: Implemented at data-access layer, not UI. watch_progress queries scoped to dashboard/resume contexts only - no export endpoints, no performance-review-shaped reports. Session tokens carry role scope; Employee sessions can only read own progress. Enforces §9 PRD constraint structurally.
- (decision) AD-6 Real-Time Dashboard Updates: Server-Sent Events (SSE) chosen over WebSocket or polling. One-directional push (server→client) matches the requirement pattern. Client establishes SSE connection on dashboard mount; server pushes assignment row updates as watch_progress writes arrive. Graceful degradation: stale connection shows refresh prompt, not silent staleness. 30-second update latency target per FR-11.
- (decision) AD-7 YouTube Integration via Adapter: YouTube IFrame API wrapped behind VideoPlayerAdapter interface. Polling-based capture (getCurrentTime every 5-10s) + onStateChange hooks. sendBeacon flush on visibilitychange/beforeunload for tab-close reliability. Adapter isolates YouTube-specific polling from watch_progress module so future Vimeo swap (event-driven timeupdate) is a drop-in replacement, not a rewrite.
- (decision) AD-8 Vector Search Architecture: pgvector extension in PostgreSQL, not dedicated vector DB. Filter-then-rank pattern: skill tag metadata pre-filter, then cosine similarity ranks within filtered set. Embedding model: text-embedding-3-small (OpenAI). Batch ingestion job (scheduled, not on-demand) respects YouTube API daily quota. Pilot scale (hundreds/low thousands of content rows) doesn't justify separate vector DB operational overhead.
- (decision) AD-9 Session Management: JWT in HTTP-only/Secure/SameSite cookie (not localStorage). Tokens carry role (HR Admin | Employee) and identity (employee_id for Employee role). auth module validates on every protected request. No session invalidates immediately on sign-out. Session expiry: [ASSUMPTION: not specified in PRD/addendum - default to 8-hour session + 7-day refresh pattern or single-session-per-day depending on pilot usage patterns].
- (decision) AD-10 Anti-Spoofing Validation: Server-side plausibility checks on watch_progress writes. Reject: (1) instantaneous completion (0→100% in under video duration), (2) impossible advance rate (more than 1.2x real-time since last update), (3) session mismatch (Employee session doesn't own the Assignment). Verified provenance label only applied to writes passing all checks. Failed validation returns 400 with reason, client retries with corrected data or surfaces error.
- (decision) AD-11 Database Schema Core Tables: (1) users (id, email, role, password_hash) - covers HR Admin + Employee identities. (2) assignments (id, employee_id, skill_id, status, created_at, updated_at, hr_override) - one row per Employee×Skill. (3) skills (id, name, description). (4) content_catalog (id, title, url, content_type, skill_tags, embedding vector(1536), metadata jsonb). (5) skill_progress (id, assignment_id, video_content_id, position_seconds, duration_seconds, event_timestamp, updated_at, provenance) - conditional write target. (6) assignment_history (audit log for FR-12 overrides, not read-path critical).
- (decision) AD-12 API Route Structure: RESTful + pragmatic RPC where needed. /api/auth/* (login, logout, session-check). /api/assignments/* (CRUD, list for dashboard). /api/content/* (discovery list, match-for-skill). /api/progress/* (update position, get resume point). /api/dashboard/stream (SSE endpoint for real-time updates). /api/skills/* (list for assignment flow). All routes validate session; Employee-scoped routes enforce employee_id match from token.
- (decision) AD-13 Frontend Routing & State: React Router for SPA navigation. Protected routes wrapped in AuthGuard (redirects to /login if no session). Role-specific route guards: /dashboard → HR Admin only, /content-discovery + /watch → Employee only. Global state via React Context for session (user role/identity), tanstack/react-query for server state caching. No global assignment/progress state - query-based, cache invalidation on mutation.
- (decision) AD-14 Error Handling & Resilience: Explicit error states for all async operations (FR-1, FR-4 consequences). Assignment save failure ≠ refresh failure (separate UX, separate toasts). Video load error shows retry + fallback, not silent blank. SSE connection drop shows refresh prompt. All client writes use optimistic UI with rollback on 4xx/5xx. Server errors return structured JSON: {error: string, code: string, retryable: boolean}.
- (decision) AD-15 Deployment Architecture: [ASSUMPTION: single-server deployment for MVP pilot given zero-budget constraint and internal-only scope]. Frontend: static build served from FastAPI /static route (Vite build output). Backend: FastAPI app server (uvicorn/gunicorn). Database: PostgreSQL single instance with pgvector extension. All three colocated on one VM/container for simplicity. [NOTE: Open Question 7 from PRD - deployment target undecided, blocking risk for 2026-07-13 launch].
- (decision) AD-16 Status & Provenance Computation: Status (Not Started/In Progress/Completed) computed from watch_progress.position_seconds / duration_seconds: 0% = Not Started, 1-99% = In Progress, 100% = Completed. Provenance (Verified/Self-reported/Needs Attention/HR Override) determined by: (1) hr_override flag → HR Override, (2) provenance='verified' + updated_at within 7 days → Verified, (3) provenance='self_reported' + updated_at > 7 days → Needs Attention, (4) provenance='self_reported' + updated_at <= 7 days → Self-reported. Computed server-side in dashboard query, not stored redundantly.
- (decision) AD-17 Content Ingestion Pipeline: Scheduled batch job (cron/background worker), not on-demand. Queries YouTube Data API v3 (respects daily quota), generates embeddings via OpenAI text-embedding-3-small, upserts to content_catalog. Runs nightly or on-demand via admin trigger. [ASSUMPTION: initial seed with ~50-100 curated videos covering common skills, then organic growth via batch runs]. Failed ingestion logs error, continues with remaining items, doesn't block.
- (decision) AD-18 Testing Strategy: Unit tests for conditional-write logic (AD-4), anti-spoofing validation (AD-10), status/provenance computation (AD-16). Integration tests for assignment flow end-to-end, watch-progress capture→dashboard update cycle, SSE connection lifecycle. E2E tests (Playwright) for all three UX scenarios (UJ-1, UJ-2, UJ-3) covering happy path + error states per FR consequences. Accessibility tests via axe-core for WCAG 2.1 AA compliance.
- (decision) AD-19: Single Source of Truth for Status/Provenance Computation. Binds: dashboard query logic, prevents modules computing badges differently. Rule: Status and Provenance computed ONLY by dashboard module via database view or stored query. watch_progress writes raw position/timestamp/provenance flag only—never computes Status. assignments stores hr_override flag only—never computes Provenance. Eliminates divergence risk from independent computation.
- (decision) AD-20: JWT employee_id Semantics. Binds: JWT token structure, query patterns. Rule: HR Admin tokens use employee_id=null (JSON null, not missing key). Employee tokens use employee_id={uuid}. Queries scoped to Employee role always filter WHERE employee_id = {token.employee_id}. Queries for HR Admin role never filter by employee_id (reads all). NULL comparison handled explicitly in SQL: IS NULL for HR checks, = {uuid} for Employee checks.
- (decision) AD-21: SSE Event Payload Structure. Binds: real-time update data consistency. Rule: SSE events carry full pre-computed row data (assignment_id, employee_name, skill_name, status, provenance, last_updated), not just trigger-to-re-query. Computed server-side via AD-19 single-source before push. Eliminates read-after-write race—client receives consistent state atomically with write completion.
- (decision) AD-22: Operations Dimension Deferred to Post-Launch. Binds: observability, backup/recovery, rate limiting, operational runbooks. Rule: MVP launches with minimal ops: application logs to stdout (container runtime captures), PostgreSQL native backup (pg_dump scheduled manually), no distributed tracing, no rate limiting beyond YouTube API quota respect (AD-17). Structured logging, metrics (Prometheus/Grafana), backup automation, and rate-limit middleware deferred to post-pilot operational maturity phase.
- (event) Architecture spine finalized. 22 ADs covering paradigm, module boundaries, data ownership, write integrity, real-time updates, security, deployment, testing. Reviewer gate surfaced 4 critical divergences (Status/Provenance computation, JWT NULL semantics, SSE consistency, operations gap)—all addressed via AD-19 through AD-22. Spine ready for implementation handoff.
