- (decision) Progress module is the single OWNER of skill_progress (reads AND writes), not just writer. No raw watch-history read path exists outside coaching-shaped methods (single-assignment status; single-row drill-down). No bulk / cross-employee / export read method exists to call. Realizes coaching-only structurally at the service/repository layer. Evidence: overall-stack research (coaching-only enforced at repo/service layer, distinct query path), DD-001 privacy_constraint, PRD 9. Dashboard reads status/provenance via the Progress module, never the skill_progress table directly.
- (decision) Single derivation authority: the Progress module is the ONLY place that derives effective (Status, Provenance) for an assignment from {watch signal, self-report staleness=7d, active HR override}. No other feature (dashboard, assignment flow) computes Status/Provenance from raw columns itself. This structurally closes the divergence half of PRD Open Question 11 (Status coherence); the remaining half (does the UI surface it clearly) stays a UX-only question. Evidence: PRD FR-8/FR-10/FR-12 all feed one derivation; OQ11.
- (decision) HR Override (FR-12) is stored as a SEPARATE record — attributed, timestamped, reversible — owned by the Progress module, NOT a field-overwrite on skill_progress. Reason: FR-12 requires the override and fresh Watch Progress to coexist and both appear in drill-down, and the override stands until an HR Admin explicitly changes it. Derivation rule: an active override wins for effective Status; its Provenance is HR Override and is never merged into or shown as Verified. HR set/reverse-override actions call a Progress-module write method (a second validated write path alongside record_watch_progress), preserving single-authority. Evidence: PRD FR-12 consequences.
- (decision) Backend domain modules (FastAPI feature-domain per overall-stack research): auth/ (session establishment + role/identity scoping, FR-13/14); assignments/ (owns assignments table; FR-1/FR-2 flow); content/ (owns content_catalog + embeddings; semantic filter-then-rank matching FR-3, discovery list FR-4, scheduled ingestion batch job); progress/ (owns skill_progress + overrides; capture FR-5/6/7 + readiness derivation FR-8..12). The Readiness Dashboard is a READ-COMPOSITION owning no table: it enumerates rows via assignments and gets each row's effective (Status, Provenance) from the progress module's derivation.
- (decision) Whether effective (Status, Provenance) is computed on read or cached as denormalized columns on the assignment row (as DD-001/prototype imply) is a mechanism the Progress module solely owns and always derives — never hand-set by another feature. Left as seed/implementation detail, not a spine invariant; the single-authority rule holds either way.

- (decision) Write-path invariant: skill_progress mutates ONLY via progress.record_watch_progress(), which (1) takes the Employee identity from the authenticated session, never the request body; (2) is a conditional write ordered by CLIENT EVENT-TIMESTAMP — skip if the incoming event-time is not newer than what is stored — ordering by time, never by position magnitude, so a legitimate rewind (lower position, newer time) is accepted while a stale out-of-order write (older time) is dropped (FR-7); (3) server-side anti-spoofing: rejects position advances inconsistent with real playback (e.g. instantaneous jump toward 100%) and requires the write be tied to the caller's own real Assignment. Only writes passing all three are persisted and eligible to derive as Verified (PRD 8 Write integrity).
- (decision) Session/role gate is enforced SERVER-SIDE on every protected request via a FastAPI dependency, not at login/routing only and never client-side. It: (a) requires a valid JWT from the HttpOnly/Secure/SameSite cookie before any Assignment/Content/Watch-Progress data is served, no flash of protected content (FR-13); (b) resolves role + identity from the verified token, not request params; (c) scopes every query to that identity so an Employee can only ever reach its OWN data (FR-14) — cross-employee access impossible regardless of how the request is formed; (d) a valid session against a role it lacks gets an explicit access-denied, not an empty result. The prototype's latent employee-switch (getEmployees/setSelectedEmployee) is a hard non-goal; no endpoint exposes it (OQ12).
- (decision) content_catalog is populated by a scheduled batch ingestion job (daily cron-style), never a live per-request YouTube search (hard ~100 search.list calls/day quota). Matching (FR-3) is filter-then-rank over already-ingested rows: metadata pre-filter on skill tag, then pgvector cosine ranking within that set, with a relevance threshold below which NO recommendation is returned (honest blank beats a bad match). IVFFlat or no index at pilot scale; revisit only if catalog grows into thousands of rows.
