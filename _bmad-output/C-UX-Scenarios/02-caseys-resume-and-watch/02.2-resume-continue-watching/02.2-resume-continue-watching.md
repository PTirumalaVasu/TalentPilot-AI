## Scenario Entry Context

**User Situation:** Casey previously started watching an assigned video and left partway through; they return to TalentPilot-AI (via direct visit, notification, or the "Continue Watching" nav item) intending to pick up where they left off.

**Mental State:**
- Hope: The system remembers exactly where they stopped, so resuming feels instant and effortless — no re-finding their place.
- Worry: Having to manually scrub through the video to find their spot, or discovering their progress wasn't saved at all.

**Discovery Method:** Casey returns to TalentPilot-AI directly, or clicks "Continue Watching" in the primary navigation, or opens a notification referencing the in-progress skill.

### Real-Time Progress Tracking (Technical)

**Watch-Position Capture:**
- Every 5–10 seconds during playback, poll `getCurrentTime()` from YouTube IFrame API
- Write `{user_id, skill_id, watch_position, timestamp}` to backend
- Use conditional writes: only write if incoming timestamp is newer (prevents regression)

**Dashboard Update Trigger:**
- Every write from Casey's player triggers backend event that updates Rita's dashboard row
- Rita sees: `Verified · [watch%] watched, [time] ago` (auto-updating)

**Tab Close Handling:**
- On tab close/unload, use `sendBeacon` to flush final position
- Ensures capture even on abrupt exit (browser crash, tab close, etc.)

**No Manual Sync:**
- Zero UI for "save progress" or "sync status"
- All tracking happens silently in background

## On-Page Interactions (Storyboard Items)

**Interaction 1: Continue Watching Card Appears**
- Casey opens TalentPilot-AI (or receives a notification: "Continue watching: Data Visualization Fundamentals — 14 min remaining").
- The assignment card is now tagged as "Continue Watching" with progress bar and "Resume at 14:32" button visible.
- **Key UX moment:** The system remembered exactly where Casey left off. No manual time-seeking needed. This is the Netflix/Spotify promise delivered.

**Interaction 2: Hover on Progress Bar (Optional)**
- Casey hovers over the progress bar and sees a tooltip: "14:32 of 28:00 — Last watched 3 days ago."
- Reinforces the exact timestamp and how long ago they paused.

**Interaction 3: Click Resume Button (Exit Action)**
- Casey clicks [▶ Resume at 14:32].
- Video player opens, video starts playing at exactly 14:32 / 28:00.
- **Key UX moment:** No loading delay, no seeking. The player knows the exact frame to start from.

**Interaction 4: Continue Watching (Real-Time Progress Tracking)**
- As Casey watches from 14:32 onward, the system continuously polls `getCurrentTime()` every 5–10 seconds.
- Each poll updates the backend with the new watch-%.
- If Casey pauses at 17:45, the system captures that.
- If Casey closes the tab abruptly, `sendBeacon` flushes the last known position.
- If Casey watches to completion (28:00), the row updates to `Verified · 100% watched`.
- **Key UX moment:** The tracking is silent and passive. Casey never sees a "saving..." message or manual sync step. Progress just updates, as a background operation.

## Real-Time Progress Tracking Implementation

**Watch-Position Capture:**
- Polling interval: Every 5–10 seconds during playback, read `getCurrentTime()` from YouTube IFrame API.
- Write to backend: `{user_id, skill_id, watch_position, timestamp}` with conditional write (only if incoming timestamp is newer).
- On tab close: `sendBeacon` flushes the final known position, ensuring no loss on abrupt exit.

**Dashboard Update Trigger:**
- Every write from Casey's player triggers a backend event that updates Rita's dashboard row.
- Rita sees: `Verified · [watch%] watched, [time] ago` — auto-updating without her needing to refresh.

**No Manual Sync Step:**
- Casey never clicks a "save progress" button.
- Casey never sees a confirmation or modal.
- The tracking happens silently in the background.

## Design Constraints

- **Video player**: Industry-standard YouTube IFrame player. Must resume at exact timestamp without seeking lag.
- **Progress persistence**: Watch-position must survive tab close, browser restart, and multi-day gaps.
- **Real-time updates**: Rita's dashboard row should update within 30 seconds of Casey's watch-position change (no stale lag).
- **No manual intervention**: Zero UI for "sync progress" or "save status." All tracking is automatic.
- **Responsive:** Desktop-primary.
- **Performance:** Resume must begin playing within 1 second of click (no noticeable seek delay).

---

## Integration Notes for Build

- **YouTube API:** Use polling-based `getCurrentTime()` (not event-driven `timeupdate`, since YouTube IFrame API doesn't support `timeupdate`).
- **Conditional writes:** Compare incoming `timestamp` against stored `timestamp` for this (user, skill); only write if incoming is newer. Prevents stale writes from regressing progress.
- **sendBeacon on unload:** On tab close or page unload, use `sendBeacon` to POST the final position. This ensures capture even on abrupt exits (browser crash, tab close, etc.).
- **Backend event:** Each write to `skill_progress` should trigger a downstream event that updates the corresponding row in Rita's `assignments` table with new provenance label.
