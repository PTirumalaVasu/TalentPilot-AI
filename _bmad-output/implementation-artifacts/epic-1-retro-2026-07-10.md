## What Went Well

- **Every code review this epic found something real**, even after disciplined TDD and thorough dev-time testing — the review step never came back empty, which is exactly its job. Examples: Story 1.3's HR_ADMIN null-check bug, Story 1.4's case-sensitive email bug, Story 1.7's `%`-escaping `ConfigParser` bug, Story 1.8's missing `Form-error-text` primitive + untested 401-interceptor chain + sign-out unhandled-rejection bug.
- **False positives were verified and dismissed, not just assumed away.** Story 1.1's tamper-test misread by two reviewers, Story 1.7's Alembic `env.py` coupling claim, and today's "role strict binary" / "generic 401 message" claims were all checked empirically before being dropped.
- **Story 1.6 verified a real FastAPI behavior empirically** (whether constructor-level router dependencies gate routes added after `include_router()`) instead of trusting general framework knowledge — this de-risked the "gate everything in Epic 2–5" plan before it became load-bearing across the rest of the roadmap.
- **Story 1.8's decision to hand-roll shadcn primitives** instead of relying on the network-dependent CLI avoided a sandbox-hang risk and established a reusable pattern (matching shadcn's real file/import structure) for future components.
- **Today's live full-stack manual verification** — the first time this project actually ran Docker + Postgres + backend + frontend together and drove it like a real user — caught a genuine bug that mocked unit tests structurally could not have surfaced.

## What Slowed Us Down

- **Dependency versions kept drifting ahead of the architecture spine's spec** — Zod v4 (spec implied v3), Tailwind v4 (spec assumed v3-style config), react-router-dom v7 (spec said v6.x) — three separate incidents in Story 1.8 alone. Two were documented well in Dev Notes at the time; one (react-router-dom) wasn't caught until code review.
- **Sprint-status tracking drifted from actual work state more than once.** Story 1.7 was fully implemented and merged via PR #28 while `sprint-status.yaml` still showed it as `backlog` (bypassed `create-story`), requiring a retroactive correction. Story 2.1 still shows `review` in sprint-status as of this retrospective even though it's already merged via PR #32 — a known, not-yet-corrected drift outside Epic 1's own scope.
- **A developer's own "this is intentional" reasoning was overturned by review** at least once (Story 1.3's EMPLOYEE-only null-check, initially defended as "asymmetric by design," later found to be a real bug affecting HR_ADMIN too). Worth remembering as a recurring risk, not a one-off.
- **Story 1.8 shipped with no live-browser verification loop for most of its life** — its own Dev Notes admit the color-token/theme mapping was simplified specifically because "no browser was available to visually verify" at implementation time. This is the root cause behind both of today's findings (see below).

## Key Discovery: Today's Live-Testing Catch

A patch applied during Story 1.8's code review (`Login.tsx` reading `location.state.from` to return a user to a deep-linked page after login) passed its own new unit test but broke live: after signing out, the *next* login — regardless of role — landed on the *previous* session's page. Root cause: `AuthProvider` wraps `BrowserRouter` in `App.tsx`, so a sign-out's state update triggers a re-render where `RequireAuth` (still transiently mounted) fires its own competing redirect with stale `state`, before the router's own location update is visible in that render pass — a structural cross-context timing race, not a call-order bug (confirmed by testing both orders live). The feature was reverted rather than chased further, since it was a low-severity nice-to-have to begin with. Full detail: `_bmad-output/implementation-artifacts/1-8-login-screen-ui.md`, "Live manual verification" note.

**This is now the second time in this project a mocked-navigate unit test passed while real router timing broke** (the first being the AC5/Task-6 interceptor-chain gap the same review pass fixed). Cross-context timing races (auth state above router state) are a structural blind spot for unit tests with mocked `useNavigate` / isolated component rendering — only a real, full-stack, browser-driven pass surfaces them.
