# Story 3.6: Cancel Assignment Flow Leaves No Orphaned Record

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **HR Admin**,
I want to cancel the assignment flow at any step,
so that I can start over or abandon the action without polluting the database.

## Acceptance Criteria

**⚠️ Read this before writing a single line of code: the core behavior this story specifies is already built and already covered by scattered tests — verified below. This story's real job is closing the remaining verification/coverage gaps, not building cancel-safety from scratch. Confirm the claim yourself against `AssignmentModal.tsx` before assuming it's stale.**

1. **Given** I am in the assignment modal at Step 1 (Employee select), Step 2 (Skill select), Step 3 (Content review), or the duplicate-pair interstitial ("Confirmation" in epics.md's wording — the `[View]`/`[Assign Again]` screen shown when the selected pair already has an Assignment), **when** I click `[Cancel]` or the `[X]` close button, **then** no Assignment record is created, no partial data persists in the backend, the modal closes cleanly, and I return to the dashboard exactly as before opening it. [Source: epics.md#Story 3.6, lines 1326-1332]
2. **Given** the same four points, **when** I close the modal via backdrop click or the `Escape` key (the `Dialog` primitive's other two close paths, both already wired since Story 3.4), **then** the same guarantee holds: no API call is made, no record is created. [Source: `frontend/src/components/ui/dialog.tsx` — backdrop-click and `Escape` both funnel into the same `onClose` prop as `[Cancel]`/`[X]`; epics.md's "close the modal" language covers all of the `Dialog`'s close affordances, not just the labeled button]
3. **And** this holds even if I cancel on Step 3 immediately before clicking `[Assign]` — i.e., having already fetched a duplicate-check result and a content match does not by itself create anything. [Source: epics.md#Story 3.6, line 1334]
4. **And** this holds under concurrency: cancelling while an in-flight `[Assign]` request has already been sent must not silently let that request's server-side write complete unacknowledged — the current implementation's answer is to **disable** Cancel/Back/Close entirely for the duration of that one request (see Dev Notes), which already satisfies "no orphaned record" by construction (there is no window where Cancel is clickable while a create is in flight). Confirm this remains true; do not weaken it by making Cancel clickable during an in-flight Assign. [Source: Story 3.4 Review Findings Round 1, patch 4]
5. **Backend guarantee, not just a frontend claim.** At least one test in this story must prove the "no partial data persists in the backend" clause against the real database (not a mocked API client) — open the real modal flow (or call the real endpoints directly) through Step 3, do **not** call `POST /api/assignments`, and assert zero `Assignment` rows exist for that (employee_id, skill_id) pair afterward. The prototype has no backend/persistence layer at all, so there is no prototype artifact to check this against — design this test fresh. [Source: user-supplied scope note — this AC has no WDS prototype equivalent]

## Dev Notes

### The central fact this story is built on: the behavior already exists

Story 3.4's own AC10 ("Cancel-safety... at any step (1, 2, 3, or the duplicate/empty/error variants)... no Assignment record is created, no partial state persists server-side") is **word-for-word this story's core requirement**, and Story 3.4's Dev Notes say so explicitly: *"'no orphaned record on cancel' (AC10 above) is nominally 3.6 but the epic's own AC wording ('at any step') makes it directly testable now, so this story implements it rather than leaving Cancel half-built."* Story 3.4's own Task 7.4 confirms the mechanism: `[Cancel]`, the `[X]` button, and `Dialog`'s backdrop-click/`Escape` handlers **all funnel through one `handleClose()` function**, which calls only `resetState()` + `onClose()` — there is no code path from any of those four triggers into `checkDuplicateAssignment`/`matchContentForSkill`/`createAssignment`. Story 3.4's review rounds also fixed the one real gap that could have violated this (disabling Cancel/Back/Close during an in-flight Assign request, so Cancel can never race a create that's already been sent).

**This mirrors the exact pattern Stories 3.2 and 3.3 already established in this project**: the epic assumed net-new work, but the real work had already landed as a side effect of an earlier story, so the "new" story's job became closing verification gaps and adding explicit, dedicated regression-test coverage — not building the feature. Treat this story the same way. Do not re-implement `handleClose()`, do not add a new "cancel" abstraction, do not touch the modal's state machine unless Task 1 turns up an actual, real gap.

### What is genuinely new work here (don't skip these just because "it's already built")

1. **Backdrop-click and `Escape` have zero dedicated test coverage today** — every existing Cancel/Close test simulates a button click. `Dialog`'s backdrop-click and `Escape` handlers were built and code-reviewed in Story 3.4 (`frontend/src/components/ui/dialog.tsx`) and share the same `onClose` prop, so they're almost certainly correct — but "almost certainly" isn't a permanent regression guard. This is Task 1.3.
2. **The duplicate-pair interstitial has zero cancel-specific test coverage.** It's a real, distinct screen state (not one of Step 1/2/3) reachable via `checkDuplicateAssignment` returning a non-empty array — epics.md's AC wording doesn't name it explicitly, but the interstitial sits between Step 2 and Step 3 in the same flow Cancel must be safe at every point of. This is Task 1.4.
3. **No test proves the backend guarantee independent of the frontend mock.** Every existing "no side effects" test asserts against a **mocked** `assignmentsApi.ts` client (`expect(createAssignment).not.toHaveBeenCalled()`) — which proves the frontend never *tries* to write, but says nothing about the backend actually being read-only end to end. Given this project's own precedent (Story 3.4 Round 2's `content_metadata` wire-key bug, which a frontend-only mocked test suite completely missed until a live HTTP call caught it), a mocked-only test suite is not sufficient evidence for a "backend guarantee" AC. This is Task 2 — the one piece of this story with no precedent anywhere in the codebase to copy from, since the WDS prototype has no backend at all to compare against.
