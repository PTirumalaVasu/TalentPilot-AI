# Story 3.1: Assignments Data Model & HR Admin Scope

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **developer**,
I want to expose the Assignments data model through Pydantic schemas and an HR-Admin-only, employee-hard-scoped service/repository layer,
so that HR Admins can create and manage Skill-to-Employee links while EMPLOYEE sessions can never read or write another employee's assignments.

## Acceptance Criteria

1. **Given** the `assignments` table already defined by Story 1.7's migration (`backend/app/assignments/models.py::Assignment`), **when** this story is implemented, **then** the model is verified — not redefined — against the epic's field list: `id` (UUID PK), `employee_id` (FK→employees), `skill_id` (FK→skills), `content_id` (nullable FK→content_catalog), `assigned_at` (timestamp, server default), `assigned_by` (FK→employees). No `models.py` edits and no new Alembic migration are expected for this AC. [Source: _bmad-output/planning-artifacts/epics.md#Story 3.1]
2. **And** `(employee_id, skill_id)` may repeat across multiple rows — the same Skill can be intentionally re-assigned to the same Employee, and each row is a distinct Assignment. No unique constraint exists or should be added on that pair.
3. **And** `assignments/schemas.py` defines `CreateAssignmentRequest` (`employee_id: UUID`, `skill_id: UUID`) and `AssignmentResponse` (full assignment fields + `status` + `provenance`, both computed server-side, never client-supplied).
4. **And** a reusable HR-Admin-only dependency exists (`auth/service.py`, alongside `get_current_user`) that rejects non-`HR_ADMIN` callers with `403 Forbidden` before any assignment mutation runs. `assignments/service.py`'s mutation-capable functions are wired through it.
5. **And** `assignments/repository.py` query functions hard-scope every `EMPLOYEE`-session read by `employee_id` in the WHERE clause (not the router/service layer), silently ignore any caller-supplied `employee_id` override for `EMPLOYEE` sessions (no error — don't reveal the scoping mechanism), and leave `HR_ADMIN` sessions unrestricted — realizing Story 1.3's AC6 forward-reference guidance now that this schema/module exists. [Source: _bmad-output/implementation-artifacts/1-3-role-and-identity-scoping-on-every-request.md]
6. **And** `AssignmentResponse.status`/`.provenance` are computed for the **trivial no-progress-yet case only**: a freshly created Assignment (no `skill_progress` row) always resolves to `status="NOT_STARTED"`, `provenance="Assigned · Awaiting first watch"`. General (Status, Provenance) derivation once watch data exists is out of scope — it belongs to `progress/` per AD-3 and does not exist until Epic 4/5.
7. **And** no HTTP route is added in this story. `assignments/router.py` keeps its existing Story 1.6 gate (`APIRouter(dependencies=[Depends(get_current_user)])`); the `POST /api/assignments` endpoint itself is Story 3.4/3.5's scope, built on top of this story's service/repository layer.

- [x] [Review][Decision] **Created Assignment rows were never actually committed.** `create_assignment` only does `session.add()` + `session.flush()`; `core/db.py::get_db()` had no commit-on-success logic at all (`async with async_session_factory() as session: yield session`, nothing after). `AsyncSession.__aexit__` does not auto-commit. The moment Story 3.4/3.5 wires a real `POST /api/assignments` route on top of `create_assignment_service` using `Depends(get_db)`, every created Assignment would silently roll back when the request session closes on success — a route that appears to work (200 response, no error) while persisting nothing. Independently confirmed by both Blind Hunter and the Acceptance Auditor; verified directly (`grep -rn "session.commit()" app/` → only `core/seeds.py`; `get_db()` had no try/commit/except). **Resolved (2026-07-10, user decision: fix `get_db()` now rather than defer):** `get_db()` now commits on successful completion and rolls back if the handler raises, establishing the app-wide commit convention — repository/service functions flush, `get_db()` commits. Zero risk to fix immediately: `get_db()` had no callers anywhere in the codebase at the time of this fix. New tests: `backend/tests/test_db.py` (`test_get_db_commits_on_successful_completion`, `test_get_db_rolls_back_on_exception`), verified via a separate session/connection that data actually persisted (not just that no exception was raised).
- [x] [Review][Patch] `uuid.UUID(current_user.user_id)` was still unguarded at both call sites (`assignments/service.py`, `assignments/repository.py`'s EMPLOYEE branch) even after Round 1's fix — that fix realigned the mock-account *data* to real UUIDs but left the parsing *code* fragile: any future non-mock auth source (real SSO) putting a non-UUID string in `CurrentUser.user_id` would reproduce the identical crash. **Fixed:** both call sites now go through a shared `_parse_user_id()` helper (`assignments/repository.py`) that raises a clean `400 INVALID_USER_ID` via `AppException` instead of an uncaught `ValueError` → 500.

Epic 2's actual ingestion/matching logic is only needed starting at Story 3.4 (content review step), not here.

- **AD-3 (single derivation authority for Status/Provenance) is binding here even though `progress/` doesn't exist yet.** This story must implement *only* the trivial default: a brand-new Assignment has no `skill_progress` row, so `status`/`provenance` are hardcoded to `"NOT_STARTED"` / `"Assigned · Awaiting first watch"` — no cross-module call needed for that case. **Do not build a general derivation engine, a `progress/` stub, or any lookup against `skill_progress`/`assignment_overrides` in this story** — that logic is `progress/`'s to own once Epic 4/5 build it, and guessing at its shape now risks contradicting whatever those stories actually define (same reasoning Story 1.3 used to defer repository hard-scoping rather than stub it). Story 1.7's review found and fixed a real `DuplicateObjectError` bug from two enum-creation paths colliding; don't reintroduce that class of bug.
- **`provenance` is a plain string, not an enum column** — per AD-3's "compute-on-read... mechanism owned privately by `progress/`; not a spine invariant" note, no `Provenance` DB column exists anywhere in the schema. Model `AssignmentResponse.provenance` as `str`, not a Pydantic/SQLAlchemy enum.

A drill-down-by-ID endpoint reached by a non-owning EMPLOYEE should 403 without confirming the record exists — but no such by-ID endpoint is built in this story (that's dashboard/drill-down territory in Epic 5); the list-query hard-scoping in Task 4 is what actually applies here.

Build the service/repository layer only.

- **Testing pattern for Task 4:** use the existing `db_session` fixture (`backend/tests/conftest.py`, live-DB, seeds pre-loaded via `run_seeds`) — the same pattern `test_database_schema.py` established. Do not build a second test-database fixture.
