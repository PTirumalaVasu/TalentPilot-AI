# Story 5.7: Delete Assignment ŌĆö Dashboard Row Action

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **HR Admin**,
I want a delete control on each Assignment row that asks me to confirm before it executes,
so that I can remove an Assignment I no longer want tracked, without risking an accidental, unconfirmed removal.

**Provenance:** this story did not exist in the original PRD/epics.md scope. It was added via `bmad-correct-course` on 2026-07-13 (`_bmad-output/planning-artifacts/sprint-change-proposal-2026-07-13.md`) alongside backend Story 3.7, after a capability gap surfaced from live use of the shipped Assignment Dashboard. This is the frontend half. Story 3.7 (backend) is **done**: `DELETE /api/assignments/{id}` already exists, is HR-Admin-only, soft-deletes (never physically removes the row), is idempotent (204 no-op on double-delete), and has no status/HR-Override restriction. This story does not touch the backend ŌĆö it wires a UI onto an endpoint that already works.

## Acceptance Criteria

1. **Given** I am viewing the Assignment Dashboard (the live grid at `/hr/dashboard`), **when** I look at any row's Actions cell, **then** I see a delete control (red bin icon) next to the existing "View Details" link, on every row regardless of Status or Provenance ŌĆö no row is exempt. [Source: epics.md Story 5.7; sprint-change-proposal-2026-07-13.md ┬¦2 "Delete eligibility by status: No restriction"]
2. **Given** I click the delete control on a row, **when** the confirmation modal opens, **then**: the delete never executes immediately ŌĆö confirmation is always required first; the modal shows the Employee name and Skill name for context; buttons are `[Cancel]` and a destructive confirm action (styled red, matching `ProvenanceDrillDownModal.tsx`'s existing `[Remove Override]`-style button).
3. **And** the confirmation copy is progress-aware (locked decision, sprint-change-proposal-2026-07-13.md ┬¦2): if the row's Status is `Not Started`, the copy is plain ("Remove this assignment?"); if the row's Status is `In Progress` or `Completed`, the copy is escalated and names the recorded signal (e.g. "This assignment has recorded progress ({X}% watched / Completed). Removing it will take it off the dashboard; the history is retained for audit."). The escalation trigger is `row.status !== "Not Started"` ŌĆö see Dev Notes for why this is the correct proxy without an extra API call.
4. **Given** I click `[Cancel]`, **when** the modal closes, **then** no request is sent, the Assignment is untouched, and the dashboard is exactly as it was.
5. **Given** I click the confirm action, **when** `DELETE /api/assignments/{id}` succeeds (204 No Content), **then**: the row disappears from the grid without a full page reload; the employee's accordion-group count and the overall `Total: {N} assignments` count both update; a success toast appears via `DashboardPage.tsx`'s existing `toastMessage`/`Toast` slot (do not create a second `Toast` instance); if the deleted row's own drill-down modal happened to be open, it closes.
6. **Given** the `DELETE` request fails (network error, unexpected 4xx/5xx), **when** the error is returned, **then** the confirmation modal shows an inline error (not a silent failure or a thrown/unhandled promise rejection), the row is NOT removed from the grid, and the HR Admin can retry (re-click confirm) or cancel.
7. **Out of scope for this story:** any restore/undo affordance for a deleted Assignment (locked decision: "One-way from the UI in this change"); any additional delete-eligibility restriction beyond what the backend already allows (Story 3.7 imposes none); any backend changes (the endpoint is done and untouched by this story).

### The progress-aware confirmation-copy trigger (AC3) ŌĆö why `status !== "Not Started"` is the right proxy

The backend's `AssignmentRow`/`DashboardResponse` (`types/dashboard.ts`) nulls `status_percentage` unless `status === "In Progress"` (Story 5.1's own established display convention, confirmed in `DrillDownResponse`'s doc comment and mirrored in the dashboard row type). This means the frontend cannot directly distinguish "Completed via real 100% watch progress" from "Completed via HR Override with zero underlying video watched" from the dashboard row alone ŌĆö that distinction only exists in the drill-down endpoint (`GET .../progress/drill-down`), which would require an extra round-trip before showing the confirmation modal.

**Decision for this story:** treat `row.status !== "Not Started"` (i.e. `"In Progress"` or `"Completed"`) as the escalation trigger, without an extra API call. This never under-triggers (any real recorded signal, verified or HR Override, gets the escalated copy) and only slightly over-triggers (a Completed-via-HR-Override-with-no-real-watch row gets escalated copy it strictly didn't need) ŌĆö an acceptable, conservative simplification for a frontend-only story that intentionally doesn't touch the backend. If `hasRecordedProgress` is `true` and `progressPercent` is `null`, the copy should say "Completed" rather than a percentage (Subtask 2.2).
