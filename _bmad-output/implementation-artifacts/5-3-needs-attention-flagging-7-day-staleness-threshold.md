# Story 5.3: Needs Attention Flagging — 7-Day Staleness Threshold

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the **system**,
I want to flag self-reported data that hasn't been updated in 7 days,
so that HR Admins can spot rows needing follow-up without manually checking every timestamp.

## Acceptance Criteria

1. **Given** a Skill Assignment whose only progress signal is a self-reported record (non-video) with a last-update timestamp, **when** `progress/` derives its Provenance, **then**: if `(now - last_update) > 7 days` → `NEEDS_ATTENTION`; otherwise → `SELF_REPORTED`. [Source: epics.md#Story 5.3, lines 1807-1811]
2. **Given** a Skill Assignment with no self-report record and no video signal at all, **then** Provenance derives as `NOT_STARTED` (the "no signal at all" branch of the same AC text). [Source: epics.md#Story 5.3, line 1811]
3. **Given** a Skill Assignment with Verified (video) progress, **then** this staleness check never applies to it — Needs Attention is exclusively a self-reported-data concern in MVP. A Verified row with an abandoned video (no new watch-position for 30+ days) is explicitly **not** flagged as Needs Attention (PRD Open Question 10, deliberately deferred). [Source: epics.md#Story 5.3, lines 1813-1815]
4. The 7-day threshold is a named config constant, not a magic number inlined at the comparison site, so it can be adjusted post-pilot without hunting through derivation logic. [Source: epics.md#Story 5.3, line 1817]
5. The derivation is a pure, stateless computation (no pre-caching, no stored "is_stale" flag) — designed to be called fresh on every read, matching AR-3's compute-on-read model. [Source: epics.md#Story 5.3, line 1819; ARCHITECTURE-SPINE.md AD-3]
6. **Confirmed scope boundary (see Dev Notes "The gap this story cannot close" below):** this story implements the derivation logic only. It does **not** build a self-report submission mechanism (endpoint, table, or UI) — none exists anywhere in this codebase or in any other story in epics.md, confirmed by direct search. Do not invent one here; that is a real, currently-unassigned product gap to flag to the user, not silently absorb into this story's scope.

## Dev Notes

### The gap this story cannot close (read before writing code)

Confirmed by direct search of the entire backend (`grep -ri "self_report\|self-report" backend/`) and of `epics.md` in full: **no self-report submission mechanism — endpoint, database column/table, or frontend UI — exists anywhere in this codebase, and no story in any epic builds one.** `skill_progress` (the only progress table that exists) has exactly two signal fields, `watch_position` and `verified`, both populated exclusively by the video-capture path (Story 4.1-4.5's YouTube Adapter + anti-spoofing). There is no code path today by which a "stored self-report record" (the phrase epics.md's own AC1 uses) could ever come into existence.

This means: **this story's derivation function is correct and fully tested, but is not reachable from any real, currently-existing user-facing code path in this MVP build.** Every Assignment in this system today either has video progress (→ `Verified`, Story 3.5/5.2's territory) or has none (→ `Not Started`) — the `Self-reported`/`Needs Attention` branch this story builds has no producer yet. This is not a flaw introduced by this story; it is a pre-existing FR-10/AR-3 requirement ("self-report staleness") whose data-producing half was never specced anywhere in the PRD or epics.md — confirmed, not assumed. **Do not attempt to build a self-report submission mechanism in this story** — it is unscoped, undesigned (no FR, no UX spec, no epic AC covers what a "self-report" submission even looks like for a `DOCUMENT`/`WEBSITE` assignment), and well beyond this story's stated AC text. Build the derivation logic exactly as specced, correctly and testably, and leave it ready for whichever future story adds the producing side.

**Recommended forward-reference, matching this project's established pattern (Story 1.3 did the same for repository hard-scoping, Story 3.4 did the same for `TODO(Story 2.4)`):** whoever eventually builds the self-report submission mechanism (or Story 5.2, if it ends up scoped to include it) should call `ProgressService.derive_self_reported_provenance()` rather than re-deriving the staleness check inline — that is the whole point of AR-3's single-derivation-authority rule.

### Why this scoping, not a bigger one

The alternative — inventing a minimal self-report table/endpoint inside this story so the derivation has something real to run against — was considered and rejected: it would mean designing a whole new FR-sized feature (what triggers a self-report? Can an Employee self-report a `VIDEO` assignment? Does it need its own UX?) with zero source-document backing, which is exactly the "scope creep past what the story asks for" this project's own conventions warn against. Story 5.3's epics.md AC text is specifically and only about the **7-day threshold comparison logic** — build that well, flag the rest.

### Precedent this story extends

`backend/app/progress/service.py::ProgressService.derive_dashboard_status_and_percent()` (Story 3.5, pulled forward from Epic 5 at the time) is the direct sibling of the function this story adds: same file, same staticmethod/pure-function shape, same "no DB access, caller passes in already-fetched values" design, same AD-3 compliance rationale. Read it before writing this story's function — match its style exactly (docstring shape, defensive-coercion tone) rather than inventing a new pattern. Note its own docstring already anticipates staleness/Provenance as separate, later work: "Still not the final Epic 5 grid: no Provenance drill-down (Story 5.2), no Needs-Attention staleness threshold (Story 5.3)..." — this story is that anticipated work.

`backend/app/content/repository.py`'s `SIMILARITY_THRESHOLD = 0.7` (Story 2.4) is the direct precedent for Task 2's config constant — a bare module-level constant, not a `Settings` field, because this value has no legitimate per-environment override case (same reasoning applies to the 7-day threshold: it is a locked PRD value — project-context.md confirms "Needs-Attention staleness threshold locked at 7 days, sourced from the original design-thinking success-metric proposal" — not an ops knob).

### Testing standards — this story is unusually cheap to test correctly

Because the derivation function takes plain `datetime` values and returns a plain enum with zero DB access, its tests need none of this project's recurring live-DB testing hazards: no `conftest.py` `db_session`/`test_engine` fixture (documented `Base.metadata.drop_all()`-wipes-the-shared-DB bug, still open per the Epic 4 retrospective's "Fix conftest.py pool corruption... critical for dashboard live-DB query tests" action item), no cross-file `asyncpg` pool-corruption pattern (Story 1.7/3.1/3.3/3.4/3.5 all hit and worked around this), no private-per-test-file engine boilerplate. Plain `pytest` functions with hand-constructed `datetime(..., tzinfo=timezone.utc)` values are sufficient and preferred — do not add unnecessary DB fixtures "to be safe."
