## Deferred from: dev-story of 3-1-assignments-data-model-and-hr-admin-scope (2026-07-10)

- **The "engine pool never disposed" item above (Story 1.7 review) predicted the wrong failure mode — the real one is worse and was hit live, not just theorized.** That item assumed the risk was a future test file *forgetting* `pytest.mark.asyncio(loop_scope="module")`. This story added two new live-DB test files (`test_assignments_repository.py`, `test_assignments_service.py`) that both correctly used the marker, and the shared `app.core.db.engine` singleton still broke — for all three live-DB files, including the pre-existing `test_database_schema.py` — the moment they ran together in one `pytest` session. The actual bug: `engine`'s pooled connections bind to whichever module-scoped loop first uses them, and **any second module-scoped-loop test file that touches the same shared pool** corrupts it (`InterfaceError`, "another operation in progress"-class failure), regardless of whether the marker is present. **Worked around in this story** by giving each new test file its own private `create_async_engine(settings.DATABASE_URL)` (see both files' module docstrings) rather than importing the shared singleton — not a fix to the underlying gap, just avoidance. `test_database_schema.py` itself was left untouched (out of this story's file-change scope) and only works today because it happens to run without a private-engine peer corrupting it first in the current file order; that's fragile, not fixed. **A real fix needs one of:** an `engine.dispose()` call between test modules (a `pytest` session/module-scoped hook, not an async pytest-asyncio fixture — those were tried and rejected here, see below), or migrating all live-DB tests to the same private-engine-per-file pattern this story used, or a proper app-lifespan-scoped engine with disposal. Whoever picks this up should treat "single shared engine across multiple live-DB test files" as the actual root cause, not "someone forgot a marker."
- **A `pytest-asyncio` async-generator fixture (even one that's not the request-scoped `db_session`, just a bare per-test session fixture) is incompatible with `pytest.mark.asyncio(loop_scope="module")` in the installed `pytest-asyncio` version** [backend/tests/test_assignments_repository.py, test_assignments_service.py] — raises `AssertionError` at fixture setup (`super().setup()` in `pytest_asyncio/plugin.py`), both for a plain per-test fixture and for an `autouse=True` module-scoped teardown-only fixture (tried as a first attempt at fixing the engine-pool item above). Every live-DB test file in this codebase now works around it by opening sessions via a plain `@asynccontextmanager` helper called directly inside each test, not a pytest fixture. Not filed as a bug against the library — likely intentional in how `loop_scope` overrides fixture scoping — but worth knowing before attempting a fixture-based refactor of any live-DB test file.
