## Story

As a **developer**,
I want the application to initialize or migrate the database schema on startup,

## Acceptance Criteria

**GIVEN** a fresh PostgreSQL instance with no tables
**WHEN** the FastAPI application starts (or `alembic upgrade head` is run)
**THEN** all required tables are created in the correct order, with the correct columns, constraints, and indexes:

1. **AC1 — `accounts` table** (Story 1.4 mock credential store parity): `id` (UUID PK), `email` (unique, not null), `password_hash` (not null), `role` (enum HR_ADMIN/EMPLOYEE, not null), `created_at` (timestamptz).
2. **AC2 — `employees` table**: `id` (UUID PK), `name` (not null), `email` (unique, not null), `role` (enum, not null), `created_at` (timestamptz). Idempotent seed populates 5 employees (Rita/Casey/Morgan/Jordan/Sam).
3. **AC3 — `skills` table**: `id` (UUID PK), `name` (unique, not null), `description` (text, optional), `embedding` (pgvector 384-dim, not null), `created_at` (timestamptz). Idempotent seed populates 5 skills with real `sentence-transformers` (`all-MiniLM-L6-v2`) embeddings.
4. **AC4 — `content_catalog` table**: `id` (UUID PK), `skill_id` (FK → skills.id, not null), `title` (not null), `description`, `type` (enum VIDEO/DOCUMENT/WEBSITE, not null), `url` (not null), `embedding` (pgvector 384-dim, not null), `source` (enum YOUTUBE/MANUAL, not null), `ingested_at` (timestamptz), `metadata` (JSON). Indexes: `idx_content_skill` (skill_id), `idx_content_embedding` (ivfflat).
5. **AC5 — `assignments` table**: `id` (UUID PK), `employee_id`/`skill_id`/`assigned_by` (FKs, not null), `content_id` (FK, nullable), `assigned_at` (timestamptz). Indexes: `idx_assignments_employee`, `idx_assignments_skill`.
6. **AC6 — `skill_progress` table**: `id` (UUID PK), `assignment_id` (FK, unique, not null), `watch_position` (int, not null), `event_time` (timestamptz, not null), `verified` (bool, default false), `updated_at` (timestamptz). Indexes: `idx_progress_assignment` (unique), `idx_progress_event_time`.
7. **AC7 — `assignment_overrides` table**: `id` (UUID PK), `assignment_id`/`set_by` (FKs, not null), `set_at` (timestamptz), `reason` (text, optional), `active` (bool, default true), `override_status` (enum, not null), `reversed_at`/`reversed_by` (nullable). Indexes: `idx_overrides_assignment`, `idx_overrides_active`.
8. **AC8 — pgvector extension enabled**: `CREATE EXTENSION IF NOT EXISTS vector`.
9. **AC9 — All timestamps are `TIMESTAMP WITH TIME ZONE` (UTC)**; all PKs are UUID.
10. **AC10 — Seed data is idempotent**: re-running the seed script multiple times produces no duplicate-key errors and no row-count growth.
11. **AC11 — `alembic upgrade head` succeeds cleanly against a fresh database** using the app's own `DATABASE_URL` (not a second, independently-maintained connection string) — verified live in this pass (see Dev Agent Record); was not previously proven true.

Schema-version-mismatch fail-fast behavior (the epic's third GIVEN/WHEN/THEN block, covering a running application detecting a stale schema at startup) is **out of scope for this story** — no such startup check exists anywhere in this codebase today, and building one was not part of the original PR #28 implementation this file documents retroactively. Flagged as a gap, not silently dropped — see Dev Notes.

## Dev Notes

- **Why the original Story 1.7 completion claim ("verified in live PostgreSQL... Docker Compose") couldn't be fully trusted at face value:** `deferred-work.md` and `project-context.md` both record Docker as unreachable in every subsequent sandbox session (1.1 through 1.6), and the schema-creation method noted for the original verification was `metadata.create_all()` via a standalone script, not the Alembic path this story's own AC (and the committed `001_initial_schema.py` file) implies is the intended mechanism. This pass is the first time `alembic upgrade head` itself was actually exercised — and it failed twice before succeeding, on two independent, real bugs (see Task 6). Whatever verification happened during the original PR #28 session did not go through Alembic, or it would have hit the same `DuplicateObjectError` this pass did.
- **Enum `create_type=False` pitfall, worth remembering codebase-wide:** `sqlalchemy.Enum(..., create_type=False)` (the *generic*, dialect-agnostic type) silently accepts and discards `create_type` as an unrecognized kwarg — it is not a real parameter on that class, only on `sqlalchemy.dialects.postgresql.ENUM`. Verified empirically (`hasattr(sa.Enum(..., create_type=False), 'create_type')` → `False`). Anyone adding a new Postgres enum column anywhere else in this codebase should use `from sqlalchemy.dialects.postgresql import ENUM as PGEnum` and `PGEnum(..., create_type=False)` if the type is created explicitly elsewhere, not `sa.Enum(..., create_type=False)`.
- **`alembic.ini`'s `sqlalchemy.url` is now dead weight, deliberately.** `env.py` overwrites it at runtime via `config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)` so there is exactly one source of truth for the connection string (`backend/.env`), matching how the app itself resolves `DATABASE_URL`. The stale value still physically in `alembic.ini` is harmless but should not be "fixed" by hand-editing it back into sync in the future — that reintroduces the exact two-copies-that-can-drift bug this pass just closed.
- **Schema-version-mismatch fail-fast startup check (epic's third GIVEN/WHEN/THEN block) is not implemented anywhere in this codebase.** No commit in `git log --all` adds any such check to `main.py` or a lifespan hook. This is a real gap relative to the full epic text, not something this retroactive story file is inventing new scope to cover — flagging it here rather than silently marking AC coverage 100%. Revisit if/when this ever needs to run somewhere schema drift is a realistic operational risk (currently: local-only single-developer pilot, AR-15/AR-16 scope).
- **No engine-disposal/lifespan hook exists yet** — this was flagged as a Story 1.1 deferred item specifically pending "once real DB traffic exists" (`deferred-work.md`). Real DB traffic now exists (this story). Still not fixed here — out of this story's own scope, but the condition that was gating it is now satisfied, so it should move up in priority for whichever story next touches `main.py`.
