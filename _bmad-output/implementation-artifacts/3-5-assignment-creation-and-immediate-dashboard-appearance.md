# Story 3.5: Assignment Creation & Immediate Dashboard Appearance

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **HR Admin**,
I want to confirm an assignment and see it appear on the dashboard immediately,
so that I know it was saved and can track progress without manual refresh.

## Acceptance Criteria

1. **Given** I confirm an assignment in the modal (Story 3.4's `[Assign]` on Step 3), **when** the `POST /api/assignments` request succeeds, **then** the new Assignment row appears in the dashboard list within 1 second (NFR-L3) ŌĆö via an immediate one-shot re-fetch/insert triggered right after the POST resolves, **not** by waiting on any background polling loop (Story 5.4's future 10ŌĆō15s poll is a separate, slower mechanism for a different concern ŌĆö see Dev Notes). [Source: epics.md#Story 3.5, lines 1299-1304]
2. **Given** the new row renders, **then** its Status badge shows `Not Started` (FR-8) and its Provenance shows `Assigned ┬Ę Awaiting first watch` ŌĆö both are the trivial zero-watch-data case (no `skill_progress` row exists yet for a just-created Assignment), computed without calling into `progress/` at all. [Source: epics.md#Story 3.5, lines 1305-1306; ARCHITECTURE-SPINE.md AD-3]
3. **Given** the Assignment was created, **then** a success toast appears with exact copy `"Ō£ō Skill assigned to {Employee first name} ŌĆö {Skill name}"`, auto-dismissing after 3ŌĆō5 seconds, and the event is announced to assistive tech (`aria-live`/`role="status"`) ŌĆö not just visually rendered. [Source: epics.md#Story 3.5, line 1307; PRD addendum accessibility line ~301 (NFR-A4)]
4. **Given** the new row renders, **then** it is visually highlighted (background color or border) for 3ŌĆō5 seconds so HR can locate it without searching, then the highlight fades back to normal row styling. [Source: epics.md#Story 3.5, line 1308]
5. **Given** the Assignment is created (POST succeeds) but the dashboard's own refresh/re-fetch fails, **then** the UI shows a distinct refresh-error state with exact copy `"Assignment saved, but dashboard didn't update. [Retry]"` (AR-13, NFR-DI2) ŌĆö the Assignment itself is never lost or implied lost, and clicking `[Retry]` re-fetches the list successfully. Do **not** use the UX doc 03.2 variant copy ("Skill assigned, but the dashboard couldn't refresh. [Refresh]") ŌĆö epics.md is authoritative on exact copy per this project's established precedent. [Source: epics.md#Story 3.5, lines 1310-1312; Dev Notes below]
6. **Given** the POST itself fails (validation, DB error), **then** behavior is unchanged from Story 3.4 (already built: modal stays open, inline error, `[Cancel]` to close) ŌĆö this story does not touch that path, only confirms it still holds. [Source: epics.md#Story 3.5, line 1314; Story 3.4 AC9]
7. **Given** this story ships before Epic 5 (Story 5.1's real dashboard grid, Story 5.2's Provenance drill-down, Story 5.4's live-polling auto-update), **then** the dashboard list this story builds is an explicit minimal placeholder ŌĆö a plain table sufficient to satisfy AC1ŌĆōAC5 ŌĆö not the final grid. Every row (new or pre-existing) shows the same trivial `Not Started` / `Assigned ┬Ę Awaiting first watch` values, since no cross-assignment Status/Provenance derivation logic exists anywhere yet (that derivation is Epic 5/AD-3's job). This is a known, intentional, documented limitation ŌĆö do not attempt to build real derivation logic in this story, and do not block this story on Epic 5 existing first. [Source: Dev Notes below ŌĆö cross-epic sequencing gap]
8. **Authorization.** The new dashboard-list endpoint this story adds is reachable only by an authenticated session; results are HR-unrestricted (all Assignments org-wide) for `HR_ADMIN`, matching the existing `list_assignments_for_employee` repository function's already-built HR-unrestricted branch. [Source: `assignments/repository.py::list_assignments_for_employee` docstring; AD-6]

### The real cross-epic sequencing gap this story must not paper over

**Epic 3 (this story) is sequenced before Epic 5 in epics.md, but this story's own AC text ("row appears on the Readiness Dashboard") assumes a dashboard grid that Epic 5 (Story 5.1) owns and has not been built yet.** `frontend/src/pages/hr/DashboardStub.tsx` today is a bare placeholder: `[+ New Assignment]` button + Sign Out + a paragraph literally stating "Coming in Epic 5 (Readiness Dashboard)." On the backend, `dashboard/router.py`/`dashboard/service.py`/`dashboard/schemas.py` are mounted (`main.py:39`, prefix `/api/dashboard`) but entirely empty stubs ŌĆö zero routes registered.

This mirrors the exact pattern Story 3.4 hit with Story 2.4 (content matching) and that the 2026-07-09 implementation-readiness report flagged generally (Story 01.1.1 ŌåÆ 03.6 forward-dependency). **Resolution for this story: build a minimal, explicitly-labeled placeholder dashboard list** (Task 4) sufficient to satisfy AC1ŌĆōAC5, not the real Epic 5 grid. Story 5.1 will supersede/extend this table with real columns/pagination/badge styling; Story 5.2 will add the Provenance drill-down; Story 5.4 will add the separate 10ŌĆō15s live-polling loop for watch-progress updates on **existing** rows (a different, slower mechanism than this story's immediate one-shot refresh ŌĆö see NFR comparison below). **Do not treat this story's table as throwaway code to delete later** ŌĆö it's the real seam Epic 5 builds on, just intentionally minimal today.

**Do not build real cross-assignment Status/Provenance derivation in this story.** No function anywhere in the codebase (checked `progress/service.py`) derives `(Status, Provenance)` from `skill_progress` data yet ŌĆö that is explicitly AD-3/Epic 5's job. Apply the same trivial values to every row in the new list endpoint, not just the newest one ŌĆö this is a known limitation (an Employee's row won't show real progress even if they've started watching) to document in Completion Notes, not silently paper over as if it were correct.

### NFR-L3 (1 second) vs. Story 5.4's polling (10ŌĆō15 seconds) ŌĆö do not conflate these

The architecture spine's Consistency Conventions table specifies live dashboard row updates via **client polling** of the dashboard read endpoint, no WebSocket/SSE, at a 10ŌĆō15s interval ŌĆö that is Story 5.4's mechanism, for a *different* concern (existing rows' watch-progress changing over time). This story's AC1 requires the new row to appear within **1 second** of the POST succeeding ŌĆö far tighter than a 10-15s poll. Implement this story's "immediate appearance" as its own one-shot re-fetch (or optimistic insert) triggered directly by the `onAssigned` callback right after `POST /api/assignments` resolves ŌĆö do not wait for, depend on, or try to reuse Story 5.4's (not-yet-built) polling loop.
