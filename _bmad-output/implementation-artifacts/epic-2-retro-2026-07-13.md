# Epic 2 Retrospective: Content Discovery & Matching

## Epic Summary

**Delivery:** 6/6 stories complete (100%) -- Stories 2.1 through 2.6 (content catalog schema, embedding model integration, YouTube batch ingestion, semantic content matching, discovery grid view, empty/error states).

**Quality:** TDD followed throughout; every story went through multi-layer adversarial code review (Blind Hunter / Edge Case Hunter / Acceptance Auditor), with Stories 2.5 and 2.6 each getting an independent second review round. ~19 technical debt items logged across the epic's story reviews -- all explicitly triaged (dismissed with reason, deferred with rationale, or accepted trade-off); none silently dropped.

**Blockers encountered:** the shared `conftest.py` `db_session`/`test_engine` fixture's `Base.metadata.drop_all()`-on-teardown wiped the real dev DB repeatedly (first hit in Story 2.1, explicitly designed around in 2.4/2.5, still referenced in 2.6); intermittent Docker/live-Postgres availability.

**Production incidents:** 0 -- no production target exists (local pilot only).

**Business outcomes:** FR-3/FR-4 (content ingestion + semantic matching) realized end-to-end, including AD-7's batch-only ingestion constraint and the pgvector filter-then-rank matching design.

## What Went Well

- **Story 2.4's semantic matching logic is some of the cleanest code in the codebase** -- a single repo query, a single service function -- and both of its subtle correctness traps (cosine-distance-vs-similarity inversion, float32 boundary-precision testing) were caught and explicitly reasoned through in the story's own dev notes rather than glossed over.
- **Second-round independent code reviews earned their keep, not just as ritual.** Story 2.6's round 2 found a real regression that round 1's own `.destroy()` patch had introduced -- silently killing the watch-progress capture pipeline forever on a React StrictMode dev remount -- caught via an actual reproduction test, not code re-reading.
- **The Epic 1 retro's live-browser-smoke-pass action item visibly paid off in Story 2.5**, catching two real bugs no unit test could have found: the `ContentResponse.metadata` field leaking the DB-internal name `content_metadata` over HTTP (dormant since Story 2.1), and a misleading "0% watched" badge on never-started assignments.
- **Architecture discipline held under real pressure.** Every AD-1 exception this epic took (reading `Skill`/`ContentCatalog` directly from `content/`) was narrow, documented, and justified in writing; Story 2.5's three-module composition (`content/` + `progress/` + `assignments/`) includes an explicit "why this isn't an AD-8 violation" note rather than quietly bending the rule.
- **A stale epics.md rewrite was caught and corrected before it shipped**, not after. Story 2.5's authoring found that a day-earlier "tier-2 fix" had rewritten Epic 2's own spec into a single-assignment-card model citing a pre-pivot UX doc; it was corrected back to the PRD's actual multi-assignment grid before build, and the story key itself was renamed to reflect the fix.

## What Slowed Us Down

- **The `conftest.py` `drop_all()` DB-wipe bug shaped test-writing conventions across the whole epic** (and beyond -- also hit in 3.1/3.3/3.4/3.5/5.3) without ever being fixed. The team got good at routing around it (private per-file engines) but that's a workaround, not a fix. Still open as of today, and already flagged as a "must-fix before Epic 5 kickoff" action item in the Epic 4 retro.
- **"A checked task-list box wasn't proof of coverage" recurred four times in this epic alone**: Story 2.1's Task 4 falsely claimed seeded-data usage; Story 2.2's failure-path test asserted only a generic exception, not the correct category; Story 2.5's Task 5 claimed COMPLETED-status coverage that didn't exist; Story 2.6's Task 2 claimed `onError` callback coverage that wasn't actually tested. Same failure mode, four separate stories -- a process gap in self-certification before review catches it, not four unrelated slips.
- **A debt item flagged specifically so someone would pick it up never got picked up.** Story 2.6 (2026-07-11) pinned the pre-existing `tsc --noEmit` error count at 25, explicitly naming it a blocker for any future story's DoD claiming a clean build. As of Story 5.5's review (2026-07-13), it has grown to 45 -- three epics later, still unowned, despite being independently re-flagged at Story 3.4 and again just yesterday.
- **Real bugs only ever surfaced via live/real execution, never via mocks alone** -- true again this epic (2.2's concurrency crash needed a throwaway reproduction script; 2.5's two field-leak/badge bugs needed a live browser; 2.6's StrictMode regression needed an actual mount/unmount test). This is now a repeatedly-confirmed pattern, not a one-off risk.
