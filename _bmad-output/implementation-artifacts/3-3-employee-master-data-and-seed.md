# Story 3.3: Employee Master Data & Seed

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **developer**,
I want to verify the Employees master table is seeded correctly and precisely aligned with the mock credential store,
so that HR Admins can assign Skills to known Employees and every login resolves to a real, identity-matched Employee row.

## Acceptance Criteria

1. **Given** the MVP's local-only mock credential store, **when** the app starts for the first time, **then** a seed script populates the `employees` table with Employee records matching the mock credentials (Rita, Casey, Morgan, Jordan, Sam) — **already true today**: `backend/app/core/seeds.py::seed_employees()` (built in Story 1.7) does exactly this. This AC is verify-only; no new seed logic is expected. [Source: _bmad-output/planning-artifacts/epics.md#Story 3.3]
2. **And** each Employee has `id` (UUID), `name`, `email`, `role` (`EMPLOYEE` or `HR_ADMIN`) — confirmed present in `Employee` (`backend/app/assignments/models.py`) and populated by `seed_employees()`.
3. **And** HR Admins are seeded as Employees with role `HR_ADMIN` — confirmed: Rita is seeded with `role="HR_ADMIN"`, all others `role="EMPLOYEE"`.
4. **And** the seed is idempotent — confirmed: `seed_employees()` checks for `RITA_ID`'s existence before inserting and returns early if already seeded.
5. **And** "matching the mock credentials" is precise, not just row-count coverage: every seeded Employee's `id`, `name`'s implied email, and `role` must correspond **exactly** to `auth/repository.py`'s `_MOCK_ACCOUNTS` entry for the same person — including `id` equality, not just email/role overlap. This is the AC this story actually has real work to do on: existing test coverage (`test_database_schema.py`) only asserts loose counts (`>= 5 employees`, `>= 1 HR_ADMIN`), never per-person identity alignment.
6. **And** Open Question 9 (production HRIS/SSO roster sourcing) remains explicitly out of scope — noted, not resolved, consistent with the epic's own AC text and the PRD's OQ9.

### Review Findings

- [x] [Review][Patch] `test_seed_employees_is_idempotent_at_the_function_level` doesn't actually exercise the insert-then-no-op transition it claims to — `_seeded_session()`'s `run_seeds()` call commits (not rolls back) seed data, so by the time this test runs (after the parametrized identity tests), `RITA_ID` already exists in the shared DB. Both calls in the test hit the early-return branch; it only proves "already-seeded stays 5," not "second call is a true no-op after the first call inserts." Independently confirmed by all 3 review layers [Source: backend/tests/test_seed_employee_identity_alignment.py]. **Fixed**: test now explicitly deletes the 5 demo employee rows within its own uncommitted transaction first (rolled back at the end, restoring ambient state), so it genuinely exercises both the insert path and the no-op path regardless of test execution order.
- [x] [Review][Patch] `_DEMO_ACCOUNTS` hardcodes/duplicates the 5 emails+roles that `auth/repository.py::_MOCK_ACCOUNTS` already defines, contradicting this story's own Dev Note ("don't hand-duplicate `_MOCK_ACCOUNTS`'s contents in the test file; call the real function"). A future 6th mock account with no matching Employee would get zero coverage from this test suite since it only walks its own hardcoded list, not the real dict. **Fixed**: `_DEMO_ACCOUNTS` is now derived directly from `_MOCK_ACCOUNTS.items()`.
- [x] [Review][Patch] The "case-normalized" claim in Subtask 2.1 isn't exercised — every email in `_DEMO_ACCOUNTS` is already lowercase/trimmed, so `find_account()`'s `.strip().lower()` normalization path never actually runs during these tests. **Fixed**: added `test_find_account_normalizes_case_and_whitespace_before_lookup`, using a mixed-case/padded-whitespace email.
- [x] [Review][Patch] `test_mock_account_role_matches_seeded_employee_role` has no `account is not None` guard before subscripting (`account["role"]`), unlike its sibling test — not reachable today (parametrize list is hardcoded to match `_MOCK_ACCOUNTS`), but inconsistent and would surface a confusing `TypeError` instead of a clear assertion if that ever changed. **Fixed**: guard added.
- [x] [Review][Patch] `backend/tests/conftest.py`'s `db_session`/`test_engine` fixtures have no warning comment at the fixture definition itself about the `drop_all()` DB-wipe behavior — the only documentation is in `deferred-work.md`, which nobody reads before running `pytest`. A one-line warning docstring at the fixture doesn't fix the bug but makes it far harder to miss. **Fixed**: added a warning docstring directly to `test_engine`'s fixture definition in `conftest.py` (comment-only, no behavior change).
- [x] [Review][Patch] `deferred-work.md`'s existing "each new live-DB test file's private engine is never disposed" item (logged during Story 3.1's review) wasn't updated to add `test_seed_employee_identity_alignment.py` to the list of offenders, even though this file repeats the exact same pattern. **Fixed**: item updated in place.
- [x] [Review][Patch] Dev Agent Record's "Agent Model Used" field still has the unfilled `{{agent_model_name_version}}` placeholder. **Fixed**.
- [x] [Review][Defer] `seed_employees()`'s idempotency guard only checks `RITA_ID`'s existence, not all 5 IDs — a hypothetical partial-seed state (e.g. a prior failed transaction that inserted Rita but not the other four) would silently skip re-seeding the missing four. Pre-existing code from Story 1.7, unchanged by this diff, low likelihood (the 5 inserts are one `session.add_all()` + single `flush()`, not independently committed) — deferred, not this story's file to fix.

## Dev Notes

- **This story is verification + precision-test-coverage work, not new implementation.** `seed_employees()` (Story 1.7) already satisfies AC1–AC4 completely and has been live-verified multiple times (Stories 1.7, 1.6, 3.1 sessions all confirmed 5 employees / correct roles / idempotent). **Do not rewrite or restructure `seed_employees()`** unless Task 1's verification finds a real, specific gap against the epic AC text — if so, stop and document exactly what's missing before touching the function.
- **AC5 is the one part of this story with genuine new value.** Story 3.1's code review (2026-07-10) found and fixed a real bug: `auth/repository.py`'s mock accounts previously issued `user_id` as plain names (`"rita"`, `"casey"`, ...), completely disconnected from `core/seeds.py`'s actual `Employee.id` UUIDs — "matching the mock credentials" was true only in the loose sense (same 5 people, same emails/roles), never in the precise sense (same `id`). That gap is now closed via `backend/app/core/seed_ids.py` (canonical UUIDs both `core/seeds.py` and `auth/repository.py` import from), but **no test asserts this alignment explicitly** — `test_database_schema.py`'s existing seed tests only check row counts and role-group counts, not per-person identity. This story's real job is writing that missing assertion, which doubles as a regression guard against anyone reintroducing the exact bug Story 3.1 just fixed.
- **`core/seed_ids.py` is deliberately dependency-free** (no `sentence_transformers`/`torch`) so importing it into `auth/repository.py` doesn't slow down the auth request path — this was a specific design decision in Story 3.1's fix, not incidental. If this story's new test file needs the IDs, import from `core/seed_ids` directly (`RITA_ID`, `CASEY_ID`, etc.), not from `core/seeds` (which does pull in the heavy ML deps at import time).
- **`find_account()` (`auth/repository.py`) is the correct way to read a mock account's `user_id`/`role` for the alignment test** — don't hand-duplicate `_MOCK_ACCOUNTS`' contents in the test file; call the real function so the test breaks if the mock store's shape ever changes.
- **Open Question 9 (HRIS/SSO roster sourcing) is explicitly not this story's problem** — the epic AC itself just asks to note it, and the PRD (§ Open Questions) already tracks it as a genuinely undecided, out-of-MVP-scope item. Do not attempt to design a roster-provisioning abstraction here.
- **No models.py, router.py, or migration changes expected** — same pattern as Story 3.1. This story only adds test coverage; if `Task 1`'s verification somehow finds `seeds.py` itself needs a fix, treat that as a scope-widening discovery to flag explicitly, not something to quietly absorb.
- **Live-DB test pattern**: use the same private-engine pattern Story 3.1's `test_assignments_repository.py`/`test_assignments_service.py` established (`create_async_engine(settings.DATABASE_URL)` local to the test file, not the shared `app.core.db.engine` singleton, and not the dead `conftest.py` `db_session`/`test_engine` fixtures) — reusing the shared singleton across another module-scoped-loop test file reproduces the connection-pool corruption bug Story 3.1 hit and worked around. See that story's test files' module docstrings for the full explanation if you need it. Alternatively, this story's tests can follow `test_database_schema.py`'s own pattern (`app.core.db.engine` + `pytest.mark.asyncio(loop_scope="module")`, no private engine) **only if** this is the sole new live-DB test file added in the same session — check `deferred-work.md`'s Story 3.1 entries before deciding.
