# Story 3.4: HR Assignment Flow — Multi-Step Modal (Employee + Skill Selection)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **HR Admin**,
I want to open an assignment modal, select an Employee and Skill, and confirm,
so that I can create new assignments quickly with auto-linked AI-recommended content.

## Acceptance Criteria

1. **Given** I am authenticated as an HR Admin on the Assignment Dashboard, **when** I click `[+ New Assignment]`, **then** a modal opens with: Title "Assign a New Skill", progress indicator "Step 1 of 3", and a Close button `[X]`. [Source: epics.md#Story 3.4, lines 1090-1100]
2. **Step 1 — Select Employee. Given** the modal shows Step 1, **then** I see section title "Who should learn this?", a searchable employee dropdown ("Select employee or search..."), `[Continue to Skill Selection]`, and `[Cancel]`. **When** I type or click the dropdown, results auto-filter by name or email and each result shows name + role. **When** I select an employee and click Continue, **then** the modal advances to Step 2 and the indicator updates to "Step 2 of 3". Continue is disabled until an employee is selected. [Source: epics.md#Story 3.4, lines 1103-1125]
3. **Step 2 — Select Skill. Given** Step 2 displays, **then** I see section title "What skill?", a searchable skill dropdown ("Search for a skill or select from recommended..."), `[Review Content]`, and `[Cancel]`. Results auto-filter by name or description. **When** I select a skill and click Review Content, **then** the backend queries content matching for this skill, the modal advances to Step 3, and the indicator updates to "Step 3 of 3". Review Content is disabled until a skill is selected. [Source: epics.md#Story 3.4, lines 1128-1150]
4. **Step 3 — Review AI-Recommended Content. Given** Step 3 displays, **then** I see section title "Recommended Learning Content", label "We've found the best match for this skill:", and an auto-populated content card: thumbnail, title, source badge "YouTube", duration in minutes, `"✓ Approved"` badge, description; `[View on YouTube]` link; a read-only Assignment Summary (Employee name, Skill name, Content title "(Approved)", "Assignment Date: Today", "Status: Will be 'Assigned · Awaiting first watch'"); and a primary `[Assign]` button plus `[Cancel]`. `[Choose Different Content]` renders as a visible but non-functional/disabled link (epic marks it "optional — out of scope for MVP"; do not build a working alternates picker). [Source: epics.md#Story 3.4, lines 1153-1195]
5. **Duplicate-pair edge case. Given** the selected (Employee, Skill) pair already has an existing Assignment, **when** Step 3 would otherwise display, **then** the modal instead shows "This skill is already assigned to {Employee}. `[View]` or `[Assign Again]`" — `[Assign Again]` proceeds into Step 3 as normal (a second intentional Assignment is allowed per AC2 of Story 3.1); `[View]` may simply close the modal for this story (dashboard drill-down navigation is Epic 5's scope, not built yet — do not block this AC on it). [Source: epics.md#Story 3.4, lines 1186-1191]
6. **No-content-match edge case. Given** the selected Skill has no matching Content, **when** Step 3 displays, **then** it shows "No approved content found yet for this skill." with `[Assign without content]` enabled (creates the Assignment with `content_id=null`) and `[Choose Different Content]` visible-but-disabled per AC4. [Source: epics.md#Story 3.4, lines 1192-1195]
7. **Content-lookup failure. Given** the content-match request fails, **when** Step 3 would load, **then** an inline error shows "Couldn't load content recommendation. `[Retry]`", and Retry re-fetches. [Source: epics.md#Story 3.4, lines 1197-1199]
8. **Loading states. Given** the employee list, skill list, or content recommendation is being fetched, **then** the affected field shows a skeleton/placeholder in place of options, and `[Cancel]` remains available. [Source: epics.md#Story 3.4, lines 1210]
9. **Assign action. Given** I am on Step 3 (with or without matched content) and click `[Assign]`, **when** `POST /api/assignments` succeeds, **then** the Assignment is created (with `content_id` set to the matched content's id, or `null` if none/skipped) and the modal closes. Dashboard-refresh behavior, the success toast, and the new-row highlight are Story 3.5's scope — this story only needs to guarantee the create call succeeds and the modal closes cleanly; do not build the toast/highlight here. [Source: epics.md#Story 3.4, lines 1179-1181; Story 3.5]
10. **Cancel-safety. Given** I click `[Cancel]` or the modal's `[X]`/backdrop at any step (1, 2, 3, or the duplicate/empty/error variants), **then** no Assignment record is created, no partial state persists server-side, the modal closes, and I return to the dashboard exactly as before opening it. This holds even from Step 3 pre-`[Assign]`-click. [Source: epics.md#Story 3.6 — binding on this story per its own "at any step" wording, even though 3.6 is nominally a separate story]
11. **Accessibility & performance.** The full flow is keyboard-navigable (Tab order, arrow-key/Enter dropdown selection), the progress indicator is programmatically associated with current step (not purely visual), error/loading state changes are announced to assistive tech (`aria-live`), the `[X]` close control has an accessible name and adequate hit target, and the happy path (open → 3 steps → Assign) is achievable in well under 2 minutes of interaction (NFR-L3). [Source: epics.md#Story 3.4, lines 1266-1274]
12. **Authorization.** Every new endpoint this story adds (employee list, skill list, content match) is reachable only by an authenticated session; assignment creation is HR_ADMIN-only via the existing `require_hr_admin` dependency (Story 3.1) — an EMPLOYEE session gets `403 FORBIDDEN_NOT_HR_ADMIN`, not a silently-broken UI. [Source: Story 3.1 AC4; AD-6]
